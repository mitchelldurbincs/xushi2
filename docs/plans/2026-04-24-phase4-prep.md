**Date:** 2026-04-24
**Author:** Track E of `2026-04-24-phase3-to-phase4-cleanup.md`
**Status:** Read-only audit. No runtime code changes.

**Purpose:** Inventory what the current (Phase-3, single-agent, 1v1 Ranger) code
reuses as-is for Phase 4 (MAPPO / CTDE / 3v3), what needs extension, what needs
rewrite, what's net-new. Then spec the VectorEnv interface, sketch the MAPPO
data flow, and conclude with a Phase-4 kickoff order.

Phase 4 target picture (from `docs/rl_design.md` §§2–3 and Phase 3→4 cleanup
plan): multi-agent PPO with centralized critic / decentralized actor, 3v3
Rangers, parallel rollouts via a VectorEnv, self-play opponent mix.

## E2 — Centralized-critic readiness check

**Finding: `src/sim/src/critic_obs.cpp` is already team-level and Phase-4 ready.
This is a significant head start.**

The C++ critic obs builder is `build_critic_obs_phase1(sim, team_perspective,
out_buffer, out_capacity)` declared in `src/sim/include/xushi2/sim/obs.h`. Key
observations from reading the two files:

1. **It takes `team_perspective`, not `agent_slot`.** The signature is already
   team-scoped — exactly what MAPPO-CTDE wants. The actor builder, by contrast,
   takes `agent_slot` and is per-agent.
2. **It contains privileged team-level fields:** world-frame positions and
   velocities of both teams' Rangers, raw objective tick counters, raw tick,
   and match seed bits. The header's contract comment explicitly says the
   critic "may iterate full sim state; it is the centralized-training side. It
   MUST NOT be called from any code path that builds an actor obs." The
   separation is already enforced by convention.
3. **Dimension is fixed at 45 floats** (`kCriticObsPhase1Dim`), actor at 31
   (`kActorObsPhase1Dim`). Python mirror exists at `obs_manifest.py`.
4. **Caveat for Phase 4:** the current critic obs is sized for **1v1**. Its
   "world_own_position / world_enemy_position" pair assumes one Ranger per
   team. When 3v3 lands, this builder needs extending to emit positions /
   velocities / HP / cooldowns for **all 3 heroes per team** (6 Rangers total).
   `kCriticObsPhase1Dim` will grow; the Python manifest must track it; actor
   prefix will also need to widen (see next caveat).
5. **Actor-prefix convention note:** the first 31 floats of the critic obs are
   the actor's obs for the team's Ranger slot. Under 3v3 this prefix convention
   breaks down: which of the 3 agents' actor obs do you put in front? Two
   options: (a) keep a single "canonical" agent's actor obs in front and list
   the other two as extra blocks, (b) drop the actor-prefix convention entirely
   for the Phase-4 critic and emit a purely team-level layout. Option (b) is
   cleaner — CTDE doesn't require the critic obs to contain any actor obs
   literally; it only requires the critic can see at least as much as any actor.

**Verdict:** The *shape* (team-scoped) is correct and reusable. The *contents*
need widening for 3v3. Renaming `build_critic_obs_phase1` → `_phase1` for the
1v1 case and adding `build_critic_obs_phase4` for the 3v3 case is the natural
evolution. This is still a big Phase-4 accelerator: the architectural split
(team-level entry point, privileged fields, actor/critic separation enforced)
is already in place and does not need to be invented.

## E3 — VectorEnv interface spec

### Signature

```python
# python/xushi2/vector_env.py  (proposed)

from typing import Any
import numpy as np

class XushiVectorEnv:
    """Batched wrapper over N independent XushiEnv instances for MAPPO.

    Shape conventions (all numpy, float32 unless noted):
      actor_obs:    (N, n_agents, actor_obs_dim)     per-agent
      critic_obs:   (N, n_teams, critic_obs_dim)     per-team (CTDE)
      action:       (N, n_agents, action_dim)
      reward:       (N, n_agents)                    agents share team reward today
      terminated:   (N,) bool                         episode-level
      truncated:    (N,) bool                         episode-level
      infos:        dict of arrays, keys like "winner" (N,), "team_a_kills" (N,)

    The learner's team is configurable per env; for 3v3 self-play we set
    n_agents = 3 (agents within one team only — opponents are not learners).
    """

    n_agents: int      # 3 at Phase 4 launch, 6 at Phase 10+
    n_teams:  int = 1  # one learner team (CTDE for a single team). Extend to 2
                       #   when we run self-play with a joint learner.
    actor_obs_dim:  int
    critic_obs_dim: int
    action_dim:     int

    def __init__(
        self,
        env_fns: list[callable],      # each returns a Phase4Env / XushiEnv
        *,
        mode: str = "sync",            # "sync" | "async"
    ) -> None: ...

    def reset(
        self,
        *,
        seeds: list[int] | None = None,
    ) -> tuple[np.ndarray, np.ndarray, dict[str, Any]]:
        """Return (actor_obs, critic_obs, infos)."""
        ...

    def step(
        self,
        actions: np.ndarray,           # (N, n_agents, action_dim)
    ) -> tuple[
        np.ndarray,  # actor_obs     (N, n_agents, actor_obs_dim)
        np.ndarray,  # critic_obs    (N, n_teams, critic_obs_dim)
        np.ndarray,  # reward        (N, n_agents)
        np.ndarray,  # terminated    (N,) bool
        np.ndarray,  # truncated     (N,) bool
        dict[str, Any],
    ]: ...

    def close(self) -> None: ...
```

Notes on the signature vs. Gymnasium's native `VectorEnv`:
- Gym's `VectorEnv.step` returns `(obs, reward, terminated, truncated, infos)`.
  We extend with a separate `critic_obs` output since actor and critic
  observation spaces genuinely differ under CTDE.
- `seeds` is a list (one per env), matching `gym.VectorEnv.reset(seed=[...])`
  semantics.
- Auto-reset on terminal should happen inside the vector env (same as
  `SyncVectorEnv`/`AsyncVectorEnv`) so the trainer never deals with a dead env.

### Which flavor: sync, async, or custom?

Context from `7ff25e7 Making async to try on my mac`: the trainer already
supports `vector_env: "sync" | "async"` and dispatches to `SyncVectorEnv` or
`AsyncVectorEnv` at `trainer.py:53`. So the async machinery exists in the
single-agent path.

**Recommendation: custom sync wrapper first, add async later.**

Reasoning:
1. The C++ sim is a cheap Python-held object (pybind11). `step_decision` is a
   C++ call; the per-env cost is dominated by sim tick work, not Python
   overhead. A `SyncVectorEnv`-style implementation is plenty fast for the 16–64
   env regime Phase 2/3 used.
2. Gym's `SyncVectorEnv` does not support our extended `(actor_obs, critic_obs)`
   return pair. Wrapping it would require per-env info-dict shuttling of the
   critic obs, which is uglier than writing a ~100-line custom wrapper.
3. Async matters once the sim tick cost gets expensive (Phase 7 fog of war,
   Phase 10+ second heroes). At that point, port the custom sync wrapper to a
   custom async variant modeled on `AsyncVectorEnv`'s pipe-based dispatch. The
   sim is trivially picklable since `MatchConfig` is a plain struct, so
   subprocess-per-env is fine.
4. **Do not use `AsyncVectorEnv` directly** for MAPPO: its per-env `step`
   returns a single obs per env. Our trainer needs actor obs and critic obs out
   of the same step — shoehorning critic obs through `infos` works but forces a
   serialize/deserialize every tick, which is exactly the async overhead we'd
   be trying to avoid.

**Short version:** custom `XushiVectorEnv` in sync mode for Phase 4 kickoff;
add an `XushiAsyncVectorEnv` once rollout collection becomes a measured
bottleneck.

## E4 — MAPPO data-flow sketch

### ASCII pipeline (per update, per env)

```
  per-env (N parallel envs)
  ┌───────────────────────────────────────────────────────────────────┐
  │  sim.step_decision(actions[3])                                     │
  │     │                                                              │
  │     ├──→ build_actor_obs  (slot 0) ─┐                              │
  │     ├──→ build_actor_obs  (slot 1) ─┼──→ actor_obs (3, 31)         │
  │     ├──→ build_actor_obs  (slot 2) ─┘                              │
  │     ├──→ build_critic_obs (team A) ────→ critic_obs (1, 45+)       │
  │     └──→ reward_calc.step() ───────────→ reward_team (scalar)      │
  └───────────────────────────────────────────────────────────────────┘
            │
            ▼
  Rollout buffer (T ticks)
  ┌───────────────────────────────────────────────────────────────────┐
  │  actor_obs   (T, N, 3, 31)     action   (T, N, 3, 6)              │
  │  critic_obs  (T, N, 1, 45+)    logprob  (T, N, 3)                 │
  │  reward      (T, N, 3)         value    (T, N, 1)    [team-value] │
  │  done        (T, N)            h_actor  (T, N, 3, H_a)            │
  │                                h_critic (T, N, 1, H_c)            │
  └───────────────────────────────────────────────────────────────────┘
            │
            ▼
  GAE (per agent, using team value V and per-agent reward r_i)
      δ_t^i = r_t^i + γ(1-done_t) V_{t+1} − V_t        # team value shared
      A_t^i = δ_t^i + γλ(1-done_t) A_{t+1}^i           # per-agent advantage
      R_t^i = A_t^i + V_t                              # per-agent return
            │
            ▼
  Minibatch iteration (segments = contiguous done-free runs, as today)
  ┌───────────────────────────────────────────────────────────────────┐
  │  Actor forward:   π_i(a_t^i | o_t^i, h_actor^i)    for i in 0..2  │
  │     → new_logprob (S, L, 3), entropy (S, L, 3)                    │
  │                                                                   │
  │  Critic forward:  V(o_t^critic, h_critic)                         │
  │     → new_value (S, L)   [one value per team per tick]            │
  │                                                                   │
  │  Policy loss   = mean over (S, L, 3) of clipped surrogate         │
  │                  using per-agent ratio r_i and per-agent adv A_i  │
  │  Value loss    = mean over (S, L)    of clipped-MSE(V, R_avg)     │
  │                  (R_avg = mean over agents of R_t^i, since all    │
  │                   agents share one team-value target)             │
  │  Entropy bonus = mean over (S, L, 3) of per-agent entropy         │
  │                                                                   │
  │  total = policy_loss + v_coef·value_loss + ent_coef·entropy_loss  │
  └───────────────────────────────────────────────────────────────────┘
```

Key invariants vs. today's code:
- `losses.py::action_logprob_and_entropy` works unchanged if we flatten the
  agent axis into the batch axis before calling it.
- `rollout_buffer.compute_gae` logic generalizes: run it on each agent's
  `(reward_i, value_shared)` slice. Current code already does
  `(N, L) → advantages (N, L)`; either loop over agents or add an agent axis.
- `trainer._ppo_minibatch_step`'s math is per-sample; masking via
  `valid_mask` and `_masked_mean` survives the agent-axis extension untouched.

### Open questions (to decide at Phase-4 kickoff, not now)

1. **Parameter sharing across Ranger slots (all 3 agents share one actor).**
   Standard MAPPO default is YES — one actor network run 3 times. Huge
   sample-efficiency win, losses nothing since all 3 Rangers are role-identical.
   **Recommended default: share.** Per-slot `log_std` or shared `log_std`
   matters less; shared is fine.
2. **Per-role heads when second heroes arrive (Phase 10+).** When Team A is
   Ranger + Scout + Tank, we'll want per-role actors but possibly a shared
   trunk. Structure Phase 4 code to factor as `{shared_trunk → per_role_head}`
   from day 1; for Phase 4 the "per_role_head" is just the one Ranger head.
3. **Self-play: opponent pool vs. frozen snapshot.** Frozen latest-checkpoint is
   the simplest starting point (each env's opponent team = policy from some
   past checkpoint, sampled uniformly from last N). Opponent pool with
   prioritized sampling (PFSP) is the upgrade. **Recommend: start with
   uniform-last-N snapshot pool. Opponent refresh on env reset, not mid-episode.**
4. **Shared team reward vs. per-agent individualized reward.** `reward.py`
   already emits per-team. Simplest Phase 4 decision: broadcast team reward to
   all 3 agents (per-agent reward tensor, all 3 entries equal). Credit
   assignment still works because each agent's advantage diverges via its own
   `h_actor` trajectory. Revisit if credit-assignment signal is too coarse.
5. **Does the critic need its own GRU?** Unclear. The privileged critic obs
   already includes positions and velocities; if the MDP feels approximately
   Markov from the critic's view, a feedforward critic suffices. Start
   feedforward and add recurrence if value-function fit is poor.

## Conclusion — Phase-4 readiness

### Already Phase-4 ready (reuse as-is)

- `RewardCalculator` in `reward.py` — already team-scoped, zero-sum
  symmetrized. Broadcast team reward to per-agent tensor and we're done.
- `runner.py` / `_build_config` — agent-count-agnostic; Phase 4 just turns on
  more sim slots.
- `losses.py` — shape-polymorphic; flattening agent axis into batch axis is all
  that's needed.
- `src/sim/src/critic_obs.cpp` architecture — team-scoped `team_perspective`
  entry point and privileged-fields section. Only the contents (1v1 → 3v3)
  widen; the structure stays.
- The `vector_env: "sync" | "async"` switch scaffolding in `trainer.py` —
  directionally right, will need a custom replacement (see below).

### Needs extension

- `env.py`, `phase3_ranger.py` — add agent axis, drop hardcoded slot constants.
- `orchestration.py` — add `phase == 4` branch in `_phase_task_spec` and in
  `_run_variant`'s trainer wiring.
- `evaluate.py` — per-agent greedy rollout; team-level win/loss aggregation
  already works.
- `src/sim/src/critic_obs.cpp` — widen to 3v3 (6 heroes' positions, velocities,
  HP, cooldowns). Bump `kCriticObsPhase1Dim` → `kCriticObsPhase4Dim`. Update
  `obs_manifest.py`.

### Needs rewrite

- `rollout_buffer.py` — add agent axis to per-agent tensors, team axis to
  critic-side tensors. Rework `iter_episode_minibatches` to pack per-agent
  slices.
- `trainer.py` (`PPOTrainer`) — becomes `MAPPOTrainer`; separate actor and
  critic forward passes; per-agent action sampling in `collect_rollout`.
- `models.py` — split `ActorCritic` into `Actor` (decentralized, recurrent) and
  `Critic` (centralized, maybe feedforward). Retain all squashed-Gaussian +
  Bernoulli sampling logic.

### Net-new

- `python/xushi2/vector_env.py` — custom `XushiVectorEnv` returning
  `(actor_obs, critic_obs, ...)` tuple. Sync-only at first.
- `python/envs/phase4_mappo.py` — 3v3 env with per-agent obs / reward, separate
  critic obs builder entry point.
- `python/train/mappo/` package — mirrors `ppo_recurrent/` layout. Config,
  trainer, buffer, orchestration, evaluate.
- `python/train/mappo/opponent_pool.py` — uniform-last-N snapshot pool for
  self-play.

### Recommended Phase-4 implementation order

1. **Widen the C++ critic obs to 3v3.** Update `critic_obs.cpp` + `obs.h` +
   `obs_manifest.py` to emit positions/velocities/HP/cooldowns for all 6 heroes
   (3 per team). Add `build_critic_obs_phase4`. Unit-test that the output is
   finite and the right shape on a fresh 3v3 sim. This is the foundation
   everything else reads from.
2. **Land a 3v3 env** (`python/envs/phase4_mappo.py`) with per-agent obs / action
   / reward and a `build_critic_obs` hook. Opponent at this stage is still
   scripted on all 3 opponent slots — no self-play yet. Write a smoke test:
   random-action rollout for 1000 ticks, no crashes.
3. **Custom `XushiVectorEnv`** (`python/xushi2/vector_env.py`), sync-only.
   Wraps N `phase4_mappo` envs; returns `(actor_obs, critic_obs, ...)` tuple.
4. **Rewrite rollout buffer** (`python/train/mappo/rollout_buffer.py`) with
   agent + team axes. Port GAE (trivial). Unit-test segment packing with a
   2-agent, 2-env toy case.
5. **Split models.py** into `Actor` + `Critic` under `train/mappo/models.py`
   (or reuse `train/models.py` by extending it). Keep squashed-Gaussian
   machinery; wire shared-actor-across-slots as the default.
6. **MAPPOTrainer** (`python/train/mappo/trainer.py`). Per-agent `collect_rollout`,
   per-team critic value. Cover with invariant tests modeled on
   `test_ppo_recurrent_invariants.py`.
7. **Wire into orchestration** — `phase == 4` branch in
   `ppo_recurrent/orchestration.py` or a new `mappo/orchestration.py`. Eval
   per-agent greedy path in `evaluate.py`.
8. **Smoke-train Phase-4 vs. scripted bots.** Sanity: does a 3v3 learner beat 3
   `walk_to_objective` bots? If yes, architecture works.
9. **Self-play** (`opponent_pool.py`): snapshot learner every K updates, sample
   uniformly from last N snapshots per env reset. Replace scripted opponents.
10. **(Later, when rollout dominates wall clock)** Port `XushiVectorEnv` to an
    async subprocess variant.

This order front-loads the low-risk plumbing (C++ obs, env, vector env,
buffer) and delays the two Phase-4-specific decisions (parameter sharing, self-
play pool policy) until we have something running to measure. Items 1–4 can
ship and be tested without a trainer; items 5–7 are the MAPPO-specific core;
items 8–10 are validation and upgrade.
